== Overview
OpenShift assumes many container images were built for permissive environments. By default it runs containers as non-root and blocks privileged ports such as 80. Images that already follow those rules—non-root user, listen on 8080 or higher—deploy cleanly. Images that expect root or port 80 fail until you fix them or accept a risky exception.

Deploy a root-friendly public httpd image, watch it fail under OpenShift defaults, then succeed with a UBI image on a high port and expose it with a Route.

== Why it matters
OpenShift does not assume your image was hardened. It applies guardrails first: random non-root UID, no bind to privileged ports. An image that ran fine on a laptop as root on port 80 hits those controls and crashes. That is useful feedback before the workload reaches production with more privilege than it needs.

Most incidents start from weak defaults, not glamorous exploits. Running non-root on high ports shrinks what an attacker can do after a compromise and makes it easier to show auditors that workloads follow least privilege.

== What does it solve
Common assumptions break on a shared cluster: root is fine, port 80 is normal, directories are owned by root, any public image is good enough. OpenShift forces an explicit choice. Either rebuild for random UIDs and high ports, or document a temporary exception. This lab shows both outcomes: Docker Hub `httpd` fails; UBI `httpd` succeeds without special SCCs.

== Your Mission
A developer just pushed a public Docker Hub `httpd` image that expects *root* and *port 80*—the same pattern attackers love after they land a foothold. Your job is to prove the platform blocks that move, then restore a hardened path that still serves traffic without widening privilege.

=== Example 1: The attacker deploys an untrusted root-friendly image (~2 min)

Simulate the break-in attempt: create a project and deploy the public Docker Hub `httpd` image. Admission may accept it, but OpenShift’s defaults should stop it from running with root-like power.

[source,sh]
----
oc new-project 101-01-httpd-demo
oc create deployment web --image=httpd
----

Wait a few seconds, then check whether the attack payload actually stuck:

[source,sh]
----
oc get pods -l app=web
oc get pods -l app=web -o wide
----

Mission check: the pod should land in `CrashLoopBackOff` or `Error` with a rising restart count—the cluster refused to hand the attacker a working root-style process.

=== Example 2: Investigate how the platform stopped the break-in (~3 min)

Collect evidence before anyone asks for `anyuid`. Logs and events are your first proof that the fail was intentional hardening, not a flaky image pull.

[source,sh]
----
oc logs -l app=web --tail=50 --all-containers=true
----

Expected failure snippet:
[source,text]
----
AH00558: httpd: Could not reliably determine the server's fully qualified domain name...
(13)Permission denied: AH00072: make_sock: could not bind to address [::]:80
(13)Permission denied: AH00072: make_sock: could not bind to address 0.0.0.0:80
no listening sockets available, shutting down
AH00015: Unable to open logs
----

Confirm with events and pod detail:

[source,sh]
----
oc get events --sort-by='.lastTimestamp' | tail -20
oc describe pod -l app=web | tail -30
----

What you just defeated: privileged port 80, root-owned log paths, and an image that assumed UID 0. OpenShift injected a random non-root UID, so the process could neither bind nor write where it wanted.

=== Example 3: Confirm the UID and port the attacker depended on (~2 min)

Inspect the runtime security context. The attacker’s image still claims port 80; the platform already forced a non-root user. That mismatch is the kill switch you want to keep.

[source,sh]
----
oc get pod -l app=web -o jsonpath='{.items[0].spec.securityContext}' ; echo
oc get pod -l app=web -o jsonpath='Container port: {.items[0].spec.containers[0].ports[0].containerPort}{"\n"}' ; echo
----

Mission check: a numeric `runAsUser` (not 0) plus container port 80. Do not "fix" this by granting extra SCC power—that is exactly the privilege the attacker wanted you to unlock.

=== Example 4: Counter with a hardened UBI image (~2 min)

Replace the break-in payload with a Red Hat UBI image built for non-root on port 8080. No SCC escalation. You restore service while denying the attacker’s privilege assumptions.

[cols="1,2",options="header"]
|===
|Defensive choice | Why it blocks the attack pattern
|Non-root user | Survives OpenShift’s random UID; shrinks post-compromise power
|High port (8080) | Removes the need for privileged binds and caps
|Curated & supported | Known provenance instead of an unvetted public tag
|===

[source,sh]
----
oc set image deployment/web *=registry.access.redhat.com/ubi9/httpd-24
oc rollout status deployment/web --timeout=120s
----

Verify the hardened workload is actually up:

[source,sh]
----
oc get pods -l app=web
oc logs -l app=web --tail=15
----

=== Example 5: Prove non-root still holds, then expose safely (~2 min)

Before publishing the app, confirm the running process is still not UID 0. Then expose it with a Route so users get service—without relaxing the UID guard that just stopped the attacker.

[source,sh]
----
oc exec deploy/web -- id
oc exec deploy/web -- id -u
----

Expect a numeric UID such as `1000690000` (not `0`). Create the Route and note the hostname:

[source,sh]
----
oc expose deployment web --port=8080
oc get route web -o jsonpath='{.spec.host}{"\n"}'
----

=== Example 6: Verify the defended service answers (~2 min)

Hit the Route and confirm the hardened image serves traffic. Success means the business still works while the privilege path the attacker needed remains closed.

[source,sh]
----
ROUTE=$(oc get route web -o jsonpath='{.spec.host}')
curl -s -o /dev/null -w "%{http_code}" "http://$ROUTE/" && echo " OK"
curl -s "http://$ROUTE/" | head -5
----

If your host cannot resolve the cluster’s ingress DNS, use a machine that can, or open the Route host in a browser:

[source,sh]
----
oc get route web -o jsonpath='{.spec.host}'
# curl -s http://<route-host>/
----

=== Example 7: Debrief — attack path vs defended path

[cols="1,2,1,1,1,2",options="header"]
|===
|Scenario | Image | Port | UID | Result | Security outcome
|Laptop local (implicit root) | docker.io/httpd:latest | 80 | 0 | Works | Attacker-friendly defaults
|OpenShift (before swap) | docker.io/httpd:latest | 80 | random | Fails | Platform blocked the break-in pattern
|OpenShift (after swap) | registry.access.redhat.com/ubi9/httpd-24 | 8080 | random | Succeeds | Service restored without privilege growth
|===

== Understanding the Attack Surface
[cols="1,2",options="header"]
|===
|Area | Risk If Misconfigured
|Root user in container | Easier path to node compromise after a breakout
|Privileged port 80 | Forces elevation or capability grants
|Fixed UID / ownership | Teams reach for `anyuid` instead of fixing the image
|Unvetted external image | Unknown CVEs and supply-chain risk
|Broad `anyuid` exceptions | Long-lived privilege that is hard to roll back
|===

== Solutions / Controls
[cols="1,2",options="header"]
|===
|Control | Purpose
|SCC restricted | Default least privilege for pods
|UBI base images | Maintained images built for OpenShift norms
|High ports | Avoid privileged bind and root
|Image scanning / signing | Know what you run and that it was not altered
|RHACS policies | Ongoing policy and runtime checks
|Digest pinning | Immutable versions instead of floating tags
|===

== Summary Table
[cols="1,2,2",options="header"]
|===
|What to Secure | Problem If Ignored | How to Secure
|Container User | Root misuse risk | Accept random non-root; avoid anyuid
|Listening Port | Privileged bind needs root | Use 8080+
|File Ownership | Crash / log failures | Pre-create + group-write dirs
|Image Provenance | Hidden vulns/backdoors | Curated UBI + scanning/signing
|Version Drift | Inconsistent rollbacks | Pin image digests
|Privilege Exceptions | Normalized risk | Track & review exceptions
|===

== FAQs
Why did it work locally?:: Local tools often allow root and port 80. OpenShift does not.
Does OpenShift distrust my image?:: It applies safe defaults until the image proves it can run without root and privileged ports.
Is granting anyuid easier?:: For a quick demo, yes. In production it becomes debt. Prefer fixing the image.
Why avoid privileged ports?:: They usually need elevated capability. High ports avoid that.
How do curated images help?:: Permissions and maintenance are already aligned with the platform.
What if I must run as root?:: Document an exception: why, who owns it, and when it expires.

== Closing Story
Prefer images that already match OpenShift defaults. Fix the image when you can; treat SCC exceptions as temporary and tracked—not the normal path.
