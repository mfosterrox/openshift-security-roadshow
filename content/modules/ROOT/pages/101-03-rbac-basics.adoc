**Duration:** ~15 minutes

== Overview
OpenShift ships useful built-in roles: `view`, `edit`, and `admin`. Learn what each can do with `oc auth can-i` and impersonation, and escalate only as far as the task requires—without writing custom Roles or handing out `cluster-admin`.

Impersonate a user bound to view, edit, and admin in turn, and use `oc auth can-i` to see what each built-in role actually allows.

== Why it matters
Access problems are usually over-permission, not clever attacks. People bind `cluster-admin` because it is faster than reading the defaults. `view`, `edit`, and `admin` already cover most team needs and give you a clear ladder: read-only, change apps, run the namespace including RoleBindings.

Using those roles speeds onboarding, keeps escalation intentional, and makes audits simpler because you are on documented platform defaults instead of one-off Role YAML.

== What does it solve
Skipping the defaults creates recurring mess:

* Temporary `cluster-admin` that never gets removed
* Custom Roles that reinvent edit or admin poorly
* Pipeline service accounts with more verbs than they need
* Confusion about whether `view` can read Secrets or `edit` can create RoleBindings

What the built-ins actually mean:

* `view` — read most namespaced objects; not Secret data; not RBAC objects
* `edit` — change Deployments, ConfigMaps, Services; cannot change RoleBindings
* `admin` — full namespace stewardship including RoleBindings; not cluster-wide ownership
* `basic-user` — self and basic project discovery
* `cluster-admin` — break-glass only; bypasses almost everything

== Your Mission
Attackers harvest over-permissioned accounts: one leaked token with `cluster-admin` ends the game. Impersonate `dev-user`, climb only through built-in `view` → `edit` → `admin`, and prove what each tier can and cannot break—without ever handing out cluster-wide power.

=== Build a sandbox the attacker would try to escalate from
[source,bash,subs="+macros,+attributes",role=execute]
----
oc new-project 101-03-r-rbac
----

=== Limit to view — block writes and Secret theft
Bind `view` and verify the attacker-shaped identity cannot deploy malware or read Secret data.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc create rolebinding dev-view --clusterrole=view --user=dev-user
oc auth can-i get pods --as=dev-user
oc auth can-i create deployment --as=dev-user
oc auth can-i get secret/mysecret --as=dev-user || true
----
Mission check: get pods = yes; create deployment = no; secret contents denied.

=== Grant edit — allow app changes, still block RBAC self-escalation
`edit` lets someone mutate workloads. It must *not* let them forge RoleBindings—that is how footholds become permanent.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete rolebinding dev-view
oc create rolebinding dev-edit --clusterrole=edit --user=dev-user
oc auth can-i create deployment --as=dev-user
oc auth can-i create rolebinding --as=dev-user
----
Mission check: create deployment = yes; create rolebinding = no.

=== Admin only when stewardship is required
Namespace `admin` is powerful inside the project (RoleBindings and Secrets). Treat it as deliberate, auditable elevation—not a default developer badge.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete rolebinding dev-edit
oc create rolebinding dev-admin --clusterrole=admin --user=dev-user
oc auth can-i create rolebinding --as=dev-user
oc auth can-i delete secret --as=dev-user
----
Mission check: both permitted *within the namespace only*.

=== Confirm the defended app is still operable
Prove legitimate work still functions under the roles you chose—security without locking the team out.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc create deployment sample --image=registry.access.redhat.com/ubi9/ubi -- sleep infinity
oc get deploy --as=dev-user
----

=== Keep the default ServiceAccount powerless
A compromised pod’s SA should not list Secrets until you explicitly bind that power. Confirm the default deny holds.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc create sa app-sa
oc auth can-i list secrets --as=system:serviceaccount:101-03-r-rbac:app-sa
----
Mission check: denied until bound.

=== Cleanup
[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete project 101-03-r-rbac --wait=false
----

== Debrief

You used built-in `view` / `edit` / `admin` at namespace scope and proved what each role can and cannot do—including that the default ServiceAccount stays powerless until you bind it.

What breaks without this:

* Over-broad RoleBindings → humans or CI wipe namespaces or steal Secrets
* ServiceAccount privilege creep → persistent foothold after a pod compromise
* Habitual `cluster-admin` → cluster-wide blast radius for routine work

Controls that matter: start with built-in roles, bind at namespace scope, keep Secrets out of `view`, and review bindings so temporary grants do not stick.

Quick facts: use `oc auth can-i` to verify; `view` cannot read Secret data; `edit` cannot change RBAC. Reserve `cluster-admin` for break-glass.
